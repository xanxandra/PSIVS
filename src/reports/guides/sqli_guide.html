SQL Injection occurs when untrusted input is interpreted as part of a SQL query. This allows attackers to alter query logic, access unauthorized data, or execute database-level actions.

Why This Happens

Applications dynamically build SQL queries using user-controlled input without proper separation between data and query structure.

Common causes:

  • String concatenation in queries
  • Improper input validation
  • Over-privileged database accounts
  • Assuming frontend validation is sufficient

Insecure Logic Pattern

User input directly modifies SQL logic:

SELECT * FROM users WHERE id = user_input

An attacker can manipulate user_input to change the query behavior.

Secure Logic Pattern

Use parameterized queries or prepared statements:

SELECT * FROM users WHERE id = ?

The database engine treats user input strictly as data, not executable logic.

Defense-in-Depth Measures

  • Always use prepared statements for database access
  • Use ORM frameworks correctly (avoid raw queries unless necessary)
  • Apply strict input validation and type enforcement
  • Limit database account privileges (no admin access for web apps)
  • Log and monitor failed queries and anomalies

What NOT to Rely On

  • Escaping input manually
  • Blacklist-based filtering
  • Frontend-only validation
  • Security through obscurity

Verification Checklist

  • All database queries are parameterized
  • No user input reaches SQL syntax positions
  • Error messages do not reveal query structure
  • Database permissions follow least privilege

Fixing SQL Injection correctly eliminates entire classes of attacks. If you fix it halfway, attackers will find the half you missed.
